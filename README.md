Synthèse du TP de ICT 301 <br><br>

<b>  I.	Single Responsibility Principle (SRP)  </b>  <br><br>
Le Single Responsibility Principle signifie que :
Une classe ne doit avoir qu’une seule responsabilité, c’est-à-dire une seule raison de changer. Autrement dit, une classe doit faire une seule chose et la faire bien.
Importance : <br>
•	Un changement n’impacte qu’une seule classe <br>
•	Le code est plus modulaire <br>
•	Les erreurs sont plus faciles à corriger <br>
•	Le refactoring est plus simple <br>

<b>  II.	Open–Closed Principle (OCP)  </b>  <br><br>
Le Open–Closed Principle dit qu’un programme doit être ouvert à l’ajout de nouvelles fonctionnalités, mais fermé à la modification du code existant.
Importance : <br>
•	Moins de bugs <br>
•	Code plus robuste <br>
•	Extensions faciles <br>
•	Maintenance simplifiée <br>

<b>  III.	LSP – Liskov Substitution Principle  </b>  <br><br>
Le Liskov Substitution Principle dit que : 
Une classe fille doit pouvoir remplacer sa classe mère sans casser le programme.

<b> IV.	ISP – Interface Segregation Principle   </b>  <br><br>
Le Interface Segregation Principle dit que :
Il vaut mieux plusieurs interfaces spécifiques qu’une seule interface générale. ISP ne force pas une classe à utiliser des méthodes inutiles

<b>  V.	DIP – Dependency Inversion Principle   </b>  <br><br>
Le Dependency Inversion Principle dit que :
Les modules de haut niveau ne doivent pas dépendre des modules de bas niveau, mais des abstractions. Il dépend des interfaces, pas des classes concrètes.
